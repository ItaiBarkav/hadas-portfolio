Overview Zoharim is a command and control system used by many different position holders in the Navy.
Ship users and shore users have completely different setups.
The setup wasn't saved for future use,
so every time they loaded it, they had to redo it.
The challenge Design a comprehensive infrastructure to support saving user settings and enable broad use across various
computer stands, while simultaneously preventing:
  • Accumulation of junk files.
  • User errors impacting different computer stands.
Research To better understand the problem, I had interviews with operational personnel. Some of the questions I asked were: Describe your setup process step by step. Is the setup dynamic? Is it common for some position holders to use the same settings? Insights: the setup is a long and tiring process. Since it isn't automatically saved, they have to redo it too often. They usually use the same settings, however operational events that take place can alter those
settings and require a different setup.
Personas Users of all types can use this feature, from novices to admins, from ship users to shore users.
This feature was born out of the same pain points for all types of users.
Novices They have minimal interaction with the system.
needs
  • Easily understand which profile is used.
  • Be aware of profile content and consequences.
mid level & admins They are well-versed in the system and can assist other users.
needs
  • Save their settings into a profile which they can reuse.
  • Easily understand which profile is used.
  • Be aware of profile content and consequences.
  • Manage profile list in their station.
Mutual Pain Points
  • Restarts reset settings, which makes setting up too repetitive.
  • Inability to use the same settings in different computer stands.
Creating the initial concept
Goals
  • Allow users in different stands to use the same settings.
  • Make it clear which settings can be saved for future use.
  • Make the setup process faster than it is today.
Concept
  • Create ready-to-use profiles that contain settings relevant to
    each position holder.
  • Create a separate setup process for profile creation,
    containing only page parts relevant to profiles,
    in which users can skip steps and only adjust the settings they want.
Advantages
  • Creating a new profile always relies on an existing profile, so you never start from scratch.
  • By creating a separate process to create a profile, it is clear which settings can be saved,
    It also gives users 1 place to manage their profile.
After presenting my concept to field personnel,
I realized the disadvantages:
Create a separate setup process. This really means duplicating pages from the system into the
profile creation process. This approach could have caused major confusion for users. It would also extend development time.
Create ready-to-use profiles relevant to each role. Unrealistic expectation. Defining each profile would have taken months of research. Users would still want to make their own profiles.
the 2nd (and final) iteration
Concept
  • Create ready-to-use profiles for 2 main ship users and 1 for shore users.
  • Incorporate the pages that already exist in the system to create a profile.
Step 1 Adjusting settings To create a profile there must be changes to
the settings.
(This prevents junk profiles from accumulating).
Step 2 choosing to create a profile An indication of changes to the current profile
is displayed. In the profile menu,
there are a few relevant options. Users who want to create a profile click the
+ Create profile button.
Step 3 verifying & saving Once the profile is loaded, a summary is displayed
with all its contents. This helps verify the settings. Modified settings are highlighted to help draw
attention to them.
Usability Testing
Goal Usability evaluation and measurement of user understanding.
I prepared 2 tasks: Create a new profile. Edit a profile.
Through these tasks, I aimed to gather insights about:
  • Can users understand the changes shown in the summary?
  • Do users understand that their editing affects other users?
Method My teammate Heni Turkian visited a base with different
types of users. She tested 5 of them and returned with the results.
Results Users understood the contents and changes of their profiles,
as well as the effects of editing. However, Heni discovered that the microcopy of some buttons
wasn't clear enough. Which made users fear clicking them. Those
issues led to modifications and retests.
Visual language Working with a design system in place,
I had a lot of visuals ready to use.
To add character to the feature I added
the following:
Geometric shapes I used geometric shapes to give the design more visual interest and dynamism.
It prevented it from feeling flat and monotonous, adding a touch of intrigue without being overwhelming.
Personalization via color The colors I picked for the profiles are balanced, muted enough for dark mode,
but noticeable enough to stand out.
